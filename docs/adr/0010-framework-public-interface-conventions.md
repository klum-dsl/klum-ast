# Framework public-interface naming and entrypoint conventions

Date: 2026-07-17

Status: Accepted

Tracking issue: [#467 — framework public-interface naming and entrypoint conventions](https://github.com/klum-dsl/klum-ast/issues/467)

Related decisions and owners:

- [ADR 0003 — Builder-first materialization](0003-builder-first-materialization.md) defines the completed-model boundary.
- [ADR 0004 — AsBuilder composition](0004-asbuilder-composition-protocol.md) and #431 own active-session Builder production.
- [ADR 0005 — generated DSL support API](0005-generated-dsl-support-api.md) and #394 own `Foo_DSL` and generated names.
- [ADR 0006 — completed Object support](0006-completed-object-support.md) and #390 own the facade implementation and migration.
- [ADR 0009 — Jackson interoperability](0009-jackson-interoperability.md) and #463 own the importer implementation.
- #474 owns generated relationship methods and their dynamic fallback behavior; #468 inventories and freezes the final JVM surface.

## Context

Builder-first 4.0 deliberately exposes a small set of framework types beside generated schema contracts, extension
interfaces, and JVM-public implementation linkage. Public visibility is necessary for some generated bytecode and Jackson
integration, but is not itself a client compatibility promise. Without a naming rule, a helper name can silently become a
stable construction API and Java-first consumers can receive a different shape from `@CompileStatic` Groovy consumers.

This ADR names framework-owned entrypoints only. It does not rename or reshape schema-authored/generated DSL methods;
those remain the responsibility of their feature owners.

## Decision

### Type and construction shape

Use UpperCamelCase type names and lower-camel method names. A public constructor is reserved for a type that callers are
expected to construct directly. `KlumAstModule` is such a type: conventional `new KlumAstModule()` is the Jackson module
registration entrypoint. A facade, adapter, or value type with no caller-owned state has no public constructor unless a
separate decision requires one.

Static factory names describe their relation to the supplied value:

| Shape | Meaning | 4.0 example |
| --- | --- | --- |
| `of(value)` | Facade over an already known, completed value; it does not construct or configure that value. | `KlumObjectSupport.of(completedObject)` |
| `using(infrastructure)` | Adapter configured by caller-owned infrastructure; it preserves rather than takes ownership of that infrastructure. | `KlumJacksonImporter.using(ObjectMapper)` / `using(ObjectReader)` |
| constructor | Direct construction is the expected framework integration. | `new KlumAstModule()` |

The sole public construction-string getter on `KlumObjectSupport` is
`getConstructionPath(): String`. It describes the immutable Builder/factory invocation path. `getModelPath()` remains the
distinct structural model path. `BreadcrumbCollector` and breadcrumb terms remain generated/runtime implementation
vocabulary; they are neither client nor extension entrypoints. #390 replaces the temporary unreleased
`getBreadcrumbPath()` facade spelling without retaining an alias.

### Root and Builder results

A root creation operation owns one complete Construction session and returns a completed DSL Object. Builder production is
explicit and is valid only inside an active session. The accepted generated spelling is Java
`Create.getAsBuilder()` and Groovy `Create.AsBuilder`; its returned Builder remains unsealed until the owning root
lifecycle completes. This terminology review changes neither the generated member set nor its descriptors.

The dynamic `KlumBuilder.link(fieldName, target)` capability supplied by #474 is a narrow supported
construction-time framework capability, not a generated relationship DSL. Its exact parameters, return shape,
relationship semantics, and diagnostics remain owned by #474. No implication is made that arbitrary public Builder
methods are client entrypoints.

### Client, extension, generated, and implementation boundaries

Every JVM-public item must be classified by #468 as exactly one of client entrypoint, extension seam, generated hook, or
implementation. This convention supplies the following default classifications:

- `KlumObjectSupport.of`, its documented getters, conventional `KlumAstModule` construction, and the #463 importer
  operations are client entrypoints when their owning issues deliver them.
- `PhaseAction`, `BuilderVisitingPhaseAction`, `ModelVisitingPhaseAction`, and `InstanceValidator` are extension types,
  never client factories. State-specific phase names communicate the Builder/Model lifecycle boundary.
- `Foo_DSL` interfaces are generated client-facing types that consumers may name but not implement; #394 owns their exact
  layout. Schema-authored DSL methods, including relationship methods, are outside this ADR.
- Generated twins, `BreadcrumbCollector`, `FactoryHelper`, `PhaseDriver`, raw companions/proxies, and internal traversal
  helpers are implementation or generated-linkage items unless their owning issue explicitly promotes a member.

An `is...` predicate is a supported API only when its owning contract explicitly makes the predicate useful to a Client
Developer. Visibility of an existing helper such as `TemplateManager.isTemplate` does not make it a compatibility promise;
#468 records its final disposition.

## Evidence and ownership

The convention was checked against the Java-first and static-Groovy fixture shapes that own the underlying behavior:

| Entry shape | Java evidence | `@CompileStatic` Groovy evidence | Delivery owner |
| --- | --- | --- | --- |
| `KlumObjectSupport.of` facade | `KlumObjectSupportSpec` compiles a Java root/subtree consumer. | The facade's generic return is directly nameable; #390 OS-3 changes that fixture to `getConstructionPath()`. | #390 |
| `Create.getAsBuilder()` / `Create.AsBuilder` | `GeneratedDslSupportSpec` compiles the Java Builder result. | `StaticAsBuilderConsumer` compiles the Groovy property spelling and concrete `Foo_DSL.Builder` result. | #394, #431 |
| `KlumJacksonImporter.using` | ADR 0009 records the reviewed Java 17 descriptors. | The same exact descriptors are accepted for static Groovy; JSON-3 must add the executable fixtures. | #463 |
| `new KlumAstModule()` | The existing public no-argument constructor is the standard Jackson-module shape. | Normal constructor invocation is statically typed; Jackson integration remains responsible for module behavior. | Jackson module |

This evidence does not turn the unimplemented importer or the pre-freeze `getConstructionPath()` correction into #467
implementation work. Their acceptance fixtures remain with their delivery owners, who must run the Groovy 3, 4, and 5
lanes when those source/API changes land.

## Consequences

- New framework APIs have a predictable Java-first construction/facade shape without treating every public helper as
  supported surface.
- Root-versus-Builder behavior is visible in the entrypoint name and return type rather than inferred from ambient state.
- #468 can classify and freeze the final public members without reopening schema-generated DSL naming or importer design.
- Client documentation may describe only delivered entrypoints. No changelog or migration entry is added for this
  decision-only record; feature owners update those records when their APIs ship.
