# Generated DSL support namespace and public Builder capabilities

Date: 2026-07-14

Status: Accepted

Tracking issue: [#394 — Make RW and other generated classes top level](https://github.com/klum-dsl/klum-ast/issues/394)

Implementation status: DSL-1, DSL-2, and DSL-G are implemented. Generated Builders use the hidden
`Foo$Builder` implementation spelling, expose `Foo_DSL.Builder extends KlumBuilder<Foo>`, and no longer carry the
`KlumRwObject` marker. DSL-3 owns remaining distribution/documentation closure. See the
[implementation plan](../implementation/adr-0005-generated-dsl-support-api.md).

Parent decision: [ADR 0003 — Builder-first materialization of DSL Objects](0003-builder-first-materialization.md)

## Context

When a schema and its consumers or tests are compiled in the same project, IntelliJ derives completion from generated
sources rather than generated class files. A full generated Builder source cannot be emitted as a second inner class of an
already existing source class. GDSL is IDE-specific and unreliable, forcing schema/model project separation is needless,
and a custom IDE plugin or LSP would be expensive and version-sensitive.

The generated surface includes more than a Builder: root factories, collection factories, and Cluster factories are also
visible to clients. A flat top-level type per generated artifact would create many collision-prone names, while the current
inner implementation names expose framework internals and the obsolete RW vocabulary.

## Decision

Generate one top-level, non-instantiable support namespace per DSL Object named `Foo_DSL`. It contains public interfaces:

```text
Foo_DSL.Builder
Foo_DSL.Factory
Foo_DSL.Builder.CollectionFactory_bars
Foo_DSL.Builder.ClusterFactory_services
```

`Foo.Create` is typed as `Foo_DSL.Factory`. A future `Foo_DSL.Template` interface is added only if Template-specific methods
actually require it. The namespace groups build-time DSL contracts; it is deliberately distinct from the runtime
`KlumObjectSupport` facade for completed objects.

The interfaces are supported as parameter, return, and receiver types in client and extension code. Clients must not
construct, subclass, or implement them. Generated implementation classes remain hidden inner/synthetic details and link to
their public interfaces through explicit generated metadata or hooks, never client-visible name parsing.

`KlumBuilder<T>` becomes a narrow public construction capability interface. The current implementation moves to an
internal/support base class. The public interface exposes only operations intended for Builder-producing protocols; raw
construction metadata, lifecycle internals, and implementation hooks are not client capabilities.

AnnoDocimal generates truthful source mirrors for `Foo_DSL` and its public nested interfaces. They describe the actual
AST-generated bytecode interfaces rather than a parallel completion-only API and include the same documentation.

These mirrors are IDE metadata only. The Klum Gradle plugin must generate them through a dedicated source-generation
lifecycle and expose their output directory to the IDE as a generated source root, but must not add that directory to any
Java/Groovy `SourceSet`, compiler input, classpath, source archive, or downstream build input. The real interfaces are
generated once by the DSL AST transformation; compiling same-named mirror sources would create a second definition and
allow bytecode and IDE contracts to diverge.

The Gradle/IntelliJ seam was deliberately left open for a focused prototype and is now resolved by DSL-G. The schema
plugin registers cacheable `createKlumDslSourceMirrors` and exposes only its output as an IDEA generated source root.
Developers run that task explicitly after schema changes; it compiles the real contract before refreshing the mirror.
The detailed task/model contract and evidence are recorded in the implementation plan. A reproducible IDE-only path is an
adoption requirement for ADR 0005, while automatic generation during a clean IDE import is not.

Canonical vocabulary uses Builder throughout. `$_RW`, `KlumRwObject`, and redundant compatibility-marker declarations are
removed before the 4.0 API is finalized.

Rename `@DelegatesToRW` to `@DelegatesToBuilder`. The old annotation is deprecated as a source-compatible alias and the AST
transformation normalizes it to the new annotation, following the `Optional` to `Validate` transformation pattern.
Generated code and documentation use only `@DelegatesToBuilder`. Applying both annotations to one parameter is a
compilation error.

## Consequences

- Same-project schema consumers receive IntelliJ completion from ordinary generated sources after running the refresh
  task.
- IDE mirrors are refreshed explicitly without participating in compilation, packaging, or downstream task inputs.
- One stable `Foo_DSL` name contains all build-time interfaces without confusing them with completed-object support.
- Java and statically compiled Groovy can name truthful factory and Builder capabilities.
- Implementation inheritance and internal hooks can evolve without widening the public Builder surface.
- The change is source-breaking for code that explicitly names `$_RW` or implements compatibility markers; migration is
  explicit in 4.0.
- All AST/signature work requires Groovy 3, 4, and 5 coverage.
- The Gradle plugin must prove clean-model registration, repeated refresh, and build-cache behavior without task cycles or
  duplicate compilation.

## Rejected alternatives

GDSL, forced project separation, a custom IntelliJ plugin, and LSP are rejected as the primary contract because they are
IDE-specific or costly and do not define a Java API. A flat set such as `Foo_Builder` and `Foo_Builder_bars` is rejected in
favor of one discoverable namespace. Keeping only a top-level Builder interface is insufficient because root, collection,
and Cluster factories are also externally visible. Naming the namespace `Foo_Support` is rejected because it collides
conceptually with completed-object `KlumObjectSupport`.

Adding the mirror directory to the normal Gradle source sets is rejected. Although that would make IDE import easy, it
would compile the IDE helper as a second public type and make a generated completion artifact affect production output.
